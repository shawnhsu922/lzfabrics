# 高端面料商城小程序 — 产品需求文档 (PRD)

## 项目背景

主体：个体工商户（营业执照登记的经营者为亲属，非本人），主营进口高端面料线上销售。
目标：搭建一个微信小程序商城，替代当前"线下成交 + Excel记库存"的模式，实现客户线上浏览、下单、支付，商家自主上传新品和管理订单。

参考项目：团队此前完成过「天长麻将」小程序（`shawnhsu922/tianchang_majiang`），沿用类似的协作模式（先写PRD文档，再交给 Codex 搭建代码骨架）。技术路线不同：麻将是实时对战场景（Node.js + WebSocket），本项目是电商场景，不需要实时通信，走标准的 REST API 即可。

## 核心功能范围（MVP）

### 客户端（小程序）

- 微信授权登录（用 openid 识别客户身份，不需要额外的账号密码注册）
- 商品分类浏览 + 搜索（材质大类如真丝/纯棉，分类不预设，由商家在后台自主创建；同时支持按商品名称关键词模糊搜索，不限于当前分类内）
- 商品列表分页加载：每页20条，支持上拉加载更多
- 商品详情页：图片、材质说明、规格参数（幅宽、克重等，后台自定义键值对添加）、单价、库存状态；页面显著位置展示色差提示文案（"图片仅供参考，色彩受光线及显示设备影响，如有疑问欢迎联系客服确认"）
- 购物车：与客户 openid 绑定并持久化保存（客户中途关闭小程序、隔天再打开，购物车内容仍在）
- 下单：
  - 计量单位为"米"，**强制整数**，不允许小数、不允许 0 或负数
  - 数量控件：`−` / `+` 按钮 + 可直接输入数字的输入框（避免大批量购买时反复点击）
  - 每款商品可单独设置**最小起订量**（如 1 米 / 2 米，因不同面料起订门槛不同，按商品单独配置而非全站统一）
- 微信支付（JSAPI 下单）
- 「我的订单」页：
  - 订单状态：待付款 → 待发货 → 已发货 → 已完成
  - 已发货状态下展示物流单号（商家手动填写，不对接顺丰/中通等物流实时查询接口）
- 联系客服：详情页/购物车等位置提供"联系客服"按钮（微信小程序原生 `open-type="contact"` 组件），客户点击后可直接在小程序内与商家实时对话，商家通过微信官方「客服小助手」小程序在手机端接收并回复消息，无需额外开发聊天系统

### 管理后台（独立网页，电脑浏览器登录）

- 商品管理：新增/编辑/上下架，上传图片，设置分类（可自由新建分类，无需预设清单）、单价、库存量（整数）、最小起订量、规格参数（自定义键值对）
- 库存管理：
  - 客户线上支付成功后，系统自动扣减对应库存
  - 后台提供手动增减库存入口（用于录入线下成交、新到货等场景，替代目前的 Excel 记账）
- 订单管理：查看订单列表/详情，标记发货状态，填写物流单号

### 明确不做（避免过度设计）

- 不对接物流公司实时查询 API
- 不做库存"预占锁定"机制防超卖（支付成功后才扣减库存；当前订单量级下，极端并发导致的超卖概率很低，出现时人工处理退款即可）
- 不做剪样与大货区分定价（统一按米数下单，后续有需要再评估）
- 不做会员等级 / 优惠券系统
- 不做多角色权限管理（后台默认单一管理员账号，你和家人共用登录）

## 数据模型（草案）

**商品 Product**
| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 主键 |
| name | string | 面料名称 |
| category | string | 材质大类，后台自建，无预设清单 |
| subcategory | string, 可空 | 二级分类，可选，后台自建 |
| description | text | 材质说明 |
| images | string[] | 图片URL列表 |
| price_per_meter | number | 单价（元/米） |
| stock_meters | integer | 当前库存（米，整数） |
| min_order_meters | integer | 最小起订量（米） |
| specs | object, 可空 | 灵活规格参数，后台自定义键值对（如幅宽、克重、成分等），无需预先定义固定字段，新增规格类型不需要改动数据库结构 |
| status | enum | 上架 / 下架 |

**购物车 Cart**
| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 主键 |
| customer_openid | string | 客户微信openid |
| items | array | [{product_id, meters}]（不锁定单价，展示时取商品当前价格） |
| updated_at | datetime | 最近更新时间 |

**订单 Order**
| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 主键 |
| customer_openid | string | 客户微信openid |
| items | array | [{product_id, name, image, meters, unit_price, subtotal}]（下单时保存商品名称与图片快照，避免商品后续被修改或下架影响历史订单展示） |
| total_amount | number | 订单总金额 |
| status | enum | 待付款/待发货/已发货/已完成/已取消 |
| tracking_number | string, 可空 | 物流单号 |
| created_at / paid_at / shipped_at | datetime | 时间戳 |

## 页面清单

**小程序端**：首页（分类入口+搜索） → 分类/搜索列表页 → 商品详情页 → 购物车 → 下单确认页（选米数）→ 支付 → 我的订单列表 → 订单详情

**管理后台**：登录页 → 商品列表/新增编辑（含分类新建、规格参数）→ 订单列表/详情（改状态、填单号）→ 库存调整入口

## 设计语言（苹果风）

- 大量留白，克制配色：米白/浅灰底色 + 一个体现"高端面料"质感的点缀色（如深灰、暗金、胭脂色），避免高饱和撞色和渐变滥用
- 字体层级分明：大字号标题 + 纤细正文，信息密度低，不拥挤
- 图片为主角：面料近拍质感图大图展示，卡片用细腻阴影而非粗边框，可考虑磨砂/毛玻璃质感的浮层
- 交互动效克制：过渡自然轻微，不要跳动感强的动画
- 组件：圆角适中不夸张，按钮扁平化

---

## 技术选型建议

- **小程序前端**：原生微信小程序（wxml/wxss/js），与麻将项目保持一致的技术栈，降低认知负担
- **后端**：Node.js + Express（REST API）
- **数据库**：MySQL 或 PostgreSQL（订单/库存/购物车是强关系型数据，比麻将项目的对局状态更适合关系型数据库）
- **支付**：微信支付 JSAPI 下单接口
- **管理后台**：轻量网页应用（Vue 或 React），不做复杂权限系统

## 分期建议

1. **Phase 1**：商品浏览 + 分类 + 搜索 + 详情页（不含支付）→ 验证：小程序能正确展示和检索所有商品
2. **Phase 2**：购物车 + 下单 + 微信支付打通 → 验证：测试环境走通一笔完整支付流程
3. **Phase 3**：库存扣减 + 订单状态流转 + 管理后台 + 客服按钮 → 验证：后台修改订单状态，小程序端能同步看到；客服按钮能正常唤起对话

## 上线全流程路线图

```
阶段A：账号与资质准备（现在就能开始，跟开发并行）
1. 小程序账号注册与认证 — 用个体工商户营业执照注册，法人本人完成微信认证（管理员须为法人本人）
2. 购买域名 — 阿里云/腾讯云都可以，完成实名认证
3. 购买云服务器 — 轻量应用服务器起步即可
4. 域名ICP备案 — 服务器买完后发起，需要营业执照+法人身份证，审核几天到约20天，
   是整个流程里最耗时的一步，建议现在就着手办理，不用等开发完成
5. 申请微信支付商户号 — 需要小程序APPID（第1步做完才有），可与备案同时进行

阶段B：开发（PRD定稿后随时可以开始，不需要等阶段A全部完成）
6. 建GitHub仓库，存放PRD文档，交给Codex开发
7. Codex 完成小程序前端、后端API、管理后台代码

阶段C：部署上线（需要阶段A的资质都办完）
8. 后端部署到云服务器，配置域名解析和HTTPS证书
9. 小程序后台「服务器域名」登记你的域名（前提：ICP备案已完成）
10. 微信支付商户号与小程序APPID绑定关联
11. 提交小程序给微信官方审核（时长以后台显示为准，通常几个工作日）
12. 审核通过，正式发布
```

**关键提醒**：第4步ICP备案最容易拖进度，因为要等第三方审核。建议现在就去买域名、买服务器、发起备案，不用等文档定稿或Codex开始写代码——**开发和资质办理这两条线完全可以同时推进，互不阻塞**。
